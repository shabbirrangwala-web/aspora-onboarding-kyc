# Aspora design system — current working plan (v1)

## Status
2026-03-17. Consolidates architecture decisions, engineer input, and color-theory exploration into an actionable plan.

---

## Context

Aspora is undergoing a rebrand. Agency Further delivered brand guidelines. The design team (Hakeeb, Suddo) is translating those into a design system while redesigning the app screen by screen. The rebrand touches every surface — some screens get token swaps, some get asset changes, some get full redesigns.

### Engineer input (Paul Lavoine, Sergei Zarochentsev — 2026-03-17)

- **iOS and Android tokens are ~90% aligned** thanks to a recent refactor by Aryan Rai. Same naming, same values through the core range.
- **Shared components are few**: button (v2), input field (v2), radio/checkbox. Everything else is either legacy or product-specific (canvas widgets, server-driven cards).
- **Android has two view systems**: Compose (new, token-aware) and XML (legacy — transfer, history, profile screens). XML screens may get fully redesigned in Compose during the rebrand.
- **Paul says don't over-engineer token sync**: "It's just a file. It takes 2 minutes." Manual reference from a repo is fine. Automated pipelines aren't worth the investment given how rarely foundations change.
- **Both platforms have dark mode infrastructure** but it's unused. Not a launch requirement — rebrand first, dark mode later.
- **Server-driven UI** has two flavors: configuration-driven (backend sends font/color strings, over-engineered, nobody updates it) and server-driven (KYC component library, canvas/IA system for home page). Paul wants configuration-driven simplified or removed.
- **Bandwidth is the constraint**. No dedicated design system implementation time. Migration happens alongside product work, screen by screen.
- **Paul's ask**: a page/catalog where developers can see all components and their states. Clear documentation of what exists.

### What color-theory established

- 8 brand anchor hexes (maroon, crimson, peach, gold, lime, teal, blue + warm neutral) — the new brand palette
- OKLCH pipeline generating 114 primitives (8 chromatic ramps x 13 steps + 2 alpha ramps x 5 steps)
- 47 semantic tokens with light/dark mode mappings (surface, on-surface, border, interactive, status colors, brand, on-brand)
- Figma plugin that syncs primitives (hidden, unpublished) and semantics (published, mode-aware aliases) to Figma variables with proper scopes
- Warm neutral ramp distinguishing from pure gray
- Perceptual uniformity via OKLCH, anchor enforcement for brand fidelity, chroma tapering at extremes

## Core principles

### Two kinds of foundational truth
- **Token-based** (color, typography, spacing, iconography): Machine-readable JSON, generated by pipelines, consumed by platforms and web projects
- **Prose-based** (photography, voice, illustration, motion): Markdown is both the source of truth and the knowledge base. No underlying artifact.

### Component model — three locations, three owners
- **uSpec in Figma**: Visual/structural spec. Anatomy, API, properties, token mapping, structure/spacing, accessibility, motion. Generated from actual component designs.
- **Design system repo**: Behavioral/usage spec. When to use, which variant for which context, do's and don'ts. Links to Figma/uSpec for visual detail.
- **Consuming project repos**: Implementation. SwiftUI, Compose, React — built by platform teams, following guidelines and uSpec specs, consuming foundation tokens.

### Manual consumption, not automated pipelines
Engineers reference `outputs/json/` and manually update their platform token files. This is fine — foundations change rarely, and the team doesn't have bandwidth for pipeline infrastructure. The design system produces well-organized reference material, not generated Swift/Kotlin.

### The design system is a reference, not an engine
It owns design decisions, not platform implementations. It produces JSON, CSS, and guidelines. Native teams consume these manually. Web projects (brand-lab, future website) consume CSS/Tailwind outputs directly.

## Deliverables

### Repo

1. **Foundations** — token JSON + guidelines for each foundation
   - Color: already established via color-theory (114 primitives, 47 semantics, light/dark)
   - Typography: new streamlined scale (header, title, label, label-heavy, body, overline) — replaces current bloated token set
   - Spacing: aligned values from native audit, clean naming
   - Corner radius: aligned values from native audit
   - Iconography, motion, photography, voice, illustration: guidelines

2. **Outputs** — generated from foundations
   - JSON: canonical token data (what engineers look at)
   - CSS: custom properties for web consumers (brand-lab, website)
   - Tailwind: theme config (if applicable)
   - AI context: deferred

3. **Workspace**
   - Pipelines: color generation (from color-theory), future typography/spacing generation
   - Figma plugin: single consolidated plugin that pushes all foundations to Figma
   - Viewer: lightweight site to see all tokens and foundations at a glance (not documentation — just a workbench)

4. **Components** — guidelines + uSpec links for shared components
   - Start with what actually exists: button, input, radio/checkbox, selector
   - Each file links to Figma/uSpec for full visual spec

5. **Migration map** — old token -> new token for the rebrand transition
   - `fillsBasePrimary500` -> `interactive/primary`
   - `textBase600` -> `on-surface/primary`
   - etc.
   - Consumed by engineers during screen-by-screen migration

### Figma

1. **Variables setup** — primitives (hidden) + semantics (published) with light/dark modes and proper scopes. Color already done via color-theory plugin. Typography, spacing, radius to follow.

2. **Old -> new mapping** — so designers can see what legacy tokens become in the new system

3. **Component specs** — generated by uSpec from actual component designs. Anatomy, API, token mapping, accessibility, motion.

## Key decisions

1. **Three repos, not a monorepo** — different owners (Hakeeb: design system, Suddo: brand-lab/docs), different workflows, different cadences
2. **Code is the source of truth, not Figma** — Figma is synced to and contributed from
3. **Manual consumption over automated pipelines** — engineers reference JSON and update platform files themselves. Paul confirmed this is fine.
4. **No component source code in design system repo** — implementations live in consuming projects
5. **The rebrand is the migration** — no separate migration workstream. Token mapping produced upfront, applied screen by screen as design touches each surface.
6. **Dark mode is phase 2** — infrastructure exists on both platforms and in color-theory's semantic mappings. Not a launch requirement.
7. **XML is dying naturally** — don't plan for it. Screens get redesigned in Compose when touched.
8. **Viewer, not documentation** — the workspace viewer is a workbench for seeing all foundations. Docs site is a separate consumer (alongside brand-lab).
9. **Single consolidated Figma plugin** — one plugin pushes all foundations, not one per foundation
10. **Server-driven UI typography contract preserved** — new token system must produce string identifiers that match backend SDU system (or coordinate removal with backend)
11. **Component scope starts small** — button, input, radio/checkbox, selector. Expand as the rebrand touches more surfaces.
