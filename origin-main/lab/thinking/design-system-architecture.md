## Background

Aspora worked with agency Further who delivered brand guidelines. Internally, the team is iterating on what works and what doesn't. Three projects exist today:

- **brand-lab** (built by Suddo): A React SPA that mixes brand exploration, documentation, and asset generation. 18 pages covering strategy, naming, visual identity, voice, art direction, photo studio, graphic device, mockups, etc. Suddo actively uses Claude Code to craft web pages and assets within this project.
- **color-theory** (built by Hakeeb): A standalone project that takes 8 brand anchor hexes, generates perceptually uniform OKLCH ramps (104 primitives, 47 semantic tokens), and syncs to Figma via a plugin. Exploration and codification happened interleaved, not sequentially.
- **docs site**: Does not exist yet. Will be built as a crafted visual experience (like wise.design), likely using Claude Code.

## Three-repo architecture

### Why not a monorepo?

- **Different owners.** Hakeeb owns the design system (tokens, pipelines, knowledge base). Suddo builds the docs site and brand-lab with Claude Code.
- **Different workflows.** Design system evolves through code explorations (branch → PR) and pipeline work. Docs site and brand-lab evolve through Claude Code generation sessions.
- **Different cadences.** Tokens change rarely. Docs site gets new pages regularly. Brand-lab changes constantly.
- **Consumer separation.** Docs site and brand-lab are consumers of the design system. Treating them that way keeps the model honest.

### The three repos

```
aspora-design-system/              ← knowledge base + tokens + guidelines + pipelines
  foundations/                       (lean, structured, AI-optimized content)
  components/                        (guidelines + Figma links — no source code)
  outputs/                           (generated: css, tailwind, figma, ios, android, ai context)
  build.ts                           (pipelines + output generation)

aspora-design-site/                ← beautiful docs experience (CONSUMER)
  (built by Suddo/team with Claude Code)
  (crafted microsites for each foundation and component)
  (consumes design system outputs for accuracy + tokens)

aspora-brand-lab/                  ← creative studio (CONSUMER)
  (asset generation, prototyping, visual exploration)
  (built with Claude Code)
  (consumes design system outputs)
```

One produces, two consume. Clean lines.

## Core concepts

### Two kinds of foundational truth
- **Token-based** (color, typography, spacing, motion): Machine-readable JSON, generated by pipelines, transformed for each platform
- **Prose-based** (photography, voice, illustration): Markdown is both the source of truth AND the knowledge base. No underlying artifact.

### Knowledge base vs. docs site
- The design system repo holds the **knowledge base** — lean, structured, AI-optimized facts. What an LLM or developer needs.
- The docs site is a **separate consumer** that presents the same knowledge beautifully — crafted microsites with interactive examples, curated galleries, polished pages. Built using Claude Code.
- Intentional duplication is okay: one for machines, one for humans. Same truths, different altitudes.

### Three kinds of exploration, three input surfaces
- **Design explorations** (Figma) → Plugin pushes to the design system repo
- **Code explorations** (scripts, pipelines) → Branch → PR into design system repo
- **Studio / generation tools** (brand-lab) → Exploration that may feed decisions back into the design system

### Component model — specs in Figma, guidelines in repo, implementations in consuming projects

Component documentation is split across three locations, each owned by the right tool:

**uspec in Figma** — the visual/structural specification. Anatomy, API, properties, color token mapping, structure/spacing, screen reader specs, motion specs. Generated by uspec directly into Figma from the actual component designs. This is the detailed "what does it look like and how is it built" documentation.

**Design system repo** — the behavioral/usage specification. When to use this component, which variant for which context, do's and don'ts, behavioral notes. Each `components/{product,editorial}/*.md` file links to the corresponding Figma/uspec spec for the full visual detail.

**Consuming project repos** — the implementation. SwiftUI, Compose, React, or whatever the platform requires. Built by the platform team, following the guidelines and uspec specs, consuming generated tokens.

AI tools get the full picture by reading the guidelines from the design system repo and following Figma links via Figma MCP to access uspec specs — anatomy, token mapping, accessibility, and motion details.

The design system repo contains **no component source code** (.tsx, .swift, .kt). It's pure knowledge: tokens, guidelines, and pointers to Figma.

### AI scope
The design system's AI concern is narrow: **make design decisions machine-readable and distributable.** It does not own or prescribe how people use AI in consuming projects. It produces AI-consumable outputs the same way it produces CSS — as a format. What Suddo does with AI in brand-lab, or what a native dev does with Copilot, is their business.

## What outputs are (and aren't)

Outputs are **token transforms and AI context** — platform-specific representations of the same values. CSS custom properties, Tailwind config, Swift constants, Android XML resources, Figma variables, plus machine-readable design system context for AI tools.

Outputs are NOT:
- Components for other platforms (native teams build their own)
- The docs site (that's a separate project)
- AI workflow guidance (that's each consumer's concern)
- Generated assets (3D renders, photo libraries, mockups — those live in brand-lab / Supabase storage)

### Assets vs. rules
The design system owns **rules** ("our photography should feel natural, eye-level, warm grading"). It does NOT own the **asset libraries** that follow those rules (500 photos, 3D renders, generated graphics). Assets are brand-lab's domain — generated by its studio tools, stored in Supabase.

## AI consumption layer

### Approach
AI context is a **build output**, generated from the same source of truth as CSS or Figma variables. Never hand-maintained.

### Distribution formats (generated by build.ts)
- `outputs/ai/claude.md` — drop into any project's CLAUDE.md
- `outputs/ai/lovable/` — .lovable/ folder ready to copy
- `outputs/ai/cursor/` — .cursor/rules/ files
- `outputs/ai/llms.txt` — hostable on docs site
- Potential: MCP server, shadcn-compatible registry

### Layered context strategy
Based on research showing focused 300 tokens often outperforms unfocused 113K tokens:
1. **Hot (~500 lines, always loaded):** Token taxonomy, naming conventions, forbidden patterns, import paths
2. **Warm (auto-attached):** Component-specific specs triggered by file glob patterns
3. **Cold (on-demand):** Full docs retrieved via MCP when agent requests

### Sync
Build-time generation solves sync. When foundations change, `build.ts` regenerates all AI outputs. Consuming projects update by pulling latest outputs.

### Open questions
- MCP server vs static files vs both?
- How to distribute updates to consuming projects?
- Registry pattern (shadcn-compatible) worth pursuing?
- Context budget per tool (Lovable's ~500 line limit vs Claude Code's larger window)?

See [ai-consumption-research.md](./ai-consumption-research.md) for full research.

## Relationship to existing projects

| Current project | What happens to it |
|-----------------|--------------------|
| **brand-lab** | Stays as its own repo. Removes foundational content (photography, voice, strategy) — that migrates to design system. Keeps studio/generation features (photo studio, graphic device, mockups, template playground). Becomes a consumer of design system outputs. |
| **color-theory** | Pipeline logic moves into design system at foundations/color/pipeline.ts. HTML deliverables may inform the docs site's color page or become a workshop tool. Standalone repo is retired. |
| **docs site** | New project. Built with Claude Code as a crafted visual experience. Consumes design system outputs. Each foundation/component gets a polished microsite page. |
| **website** | Separate project. Built in its own repo (not brand-lab). Consumes design system outputs. |

### What changes in brand-lab specifically

Brand-lab currently has 18 pages mixing exploration, documentation, and generation. In the new model:

**Stays in brand-lab (generation/studio features):**
- Photo Studio (AI photo generation)
- Graphic Device (visual effects editor)
- Mockup Studio
- Template Playground
- Copy generation tools

**Migrates to design system repo (foundational knowledge):**
- Color palette definitions → foundations/color/
- Typography pairings → foundations/typography/
- Art direction / photography philosophy → foundations/photography/guidelines.md
- Voice & messaging framework → foundations/voice/guidelines.md
- Brand pillars, positioning → strategy/

**Migrates to docs site (documentation/presentation):**
- Visual identity showcase pages
- Mood boards (as curated presentation)
- Competitor landscape, perceptual maps (as strategy documentation)
- Any page that's primarily "here's what our brand looks like"

**Likely retired (served their exploratory purpose):**
- Naming/tagline scoring (decisions have been made)
- SWOT analysis (one-time strategic exercise)
- Persona cards (migrate as content, retire the interactive tool)

**Brand-lab consumes design system outputs:**
- Replaces hand-picked hex values in index.css with generated CSS variables
- Replaces internal design-tokens.ts with imported token package
- Generation tools use real tokens instead of hardcoded values

## Key decisions made
1. **Three repos, not a monorepo** — different owners, workflows, cadences
2. Code is the source of truth, not Figma (Figma is synced to, and contributed from)
3. Token-based and prose-based foundations coexist — not everything needs a pipeline
4. Knowledge base (design system) and docs site (presentation) are separate projects
5. AI context is a build output, not a pointer into the repo
6. Outputs are token transforms + AI context — native platforms build their own components
7. Design system owns "what are our design decisions" — not "how to use AI"
8. Web pages for the actual website should be built in the website project, not brand-lab
9. Design system owns rules, not assets — generated assets (3D, photos, mockups) live in brand-lab / Supabase
10. Components split into product (functional UI) and editorial (marketing/content)
11. uspec for component specs in Figma — design system repo holds guidelines + links, not source code
12. No component source code in design system repo — implementations live in consuming projects
13. Figma serves as living extension of design system via uspec specs and MCP access
