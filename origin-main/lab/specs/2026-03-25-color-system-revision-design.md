# Color system revision — design spec

## Date

2026-03-25

## Status

Draft — pending stress-testing and team review

## Problem

Five issues with the current color system:

1. **Traffic light colors look washed.** Status colors (error, success, warning) derive from brand ramps (crimson, lime, gold). These brand hues are optimized for expression, not functional signaling. Crimson reads as pink-magenta, not danger-red. Lime goes olive at mid-steps. Gold gets muddy amber.

2. **Semantic token structure is ambiguous.** Current tokens (`base`, `subtle`, `text`, `border`) don't clearly communicate which tokens pair with each other. No explicit "text on a solid fill" token — components use the generic `on-brand/light` alpha token instead.

3. **No accent color.** `info` serves as a fourth status color, but in practice "informational" UI moments are really accent highlights. There's no dedicated accent concept.

4. **Interactive states don't scale.** Dedicated hover/pressed tokens exist only for the neutral interactive color. Colored buttons (destructive, accent) have no interaction state tokens, forcing ad-hoc solutions.

5. **Disabled tokens collide.** `interactive.primary-disabled` (neutral-300) and `on-surface.disabled` (neutral-300) alias the same primitive — zero contrast between a disabled button fill and its label.

## Decisions

### 1. New functional ramps for status colors

Introduce three new primitive ramps — `red`, `green`, `yellow` — with hues optimized for functional clarity. These are independent of the brand ramps.

| Ramp | Purpose | Target OKLCH hue | Notes |
|------|---------|------------------|-------|
| red | Error/negative signaling | ~25-29 (true red) | Distinct from crimson (~350, pink-magenta) |
| green | Success/positive signaling | ~145-155 (true green) | Distinct from lime (~130, chartreuse) |
| yellow | Warning/caution signaling | ~85-95 (clear yellow) | Distinct from gold (~80, warm amber) |

Each ramp follows the same 13-step structure (50-1000) and is generated by the existing OKLCH pipeline. Anchors will be selected for maximum saturation at the steps that status semantics need (primarily 600 for solid fills in light mode, 300 for text in dark mode).

Brand ramps (crimson, lime, gold) remain unchanged in primitives for decorative/brand use.

### 2. Rename info to accent

Drop `info` as a status color. Promote blue to a general-purpose `accent` role. In most product UI, "informational" moments are really accent highlights — a separate info status is unnecessary overhead.

The `accent` group gets the same 5-token structure as status colors (see below).

### 3. New semantic token structure for status and accent

Replace the current 4-token structure (`base`, `subtle`, `text`, `border`) with a 5-token structure that makes pairing relationships explicit:

| Token | Role | Example use |
|-------|------|-------------|
| `solid` | Strong fill background | Destructive button, solid badge |
| `light` | Tinted/subtle surface | Warning banner, error field background |
| `on-solid` | Text/icon on solid fill | White text on a red button |
| `on-light` | Text/icon on light surface or neutral surface | Error message, helper text, banner copy |
| `border` | Stroke color | Error input ring, accent-outlined button |

`on-light` doubles as a standalone text/icon color on neutral surfaces. If it's accessible on the `light` tint (~50 step), it's definitely accessible on white.

**Token mapping (light mode):**

| Group | solid | light | on-solid | on-light | border |
|-------|-------|-------|----------|----------|--------|
| error | red-600 | red-50 | neutral-50 | red-700 | red-600 |
| success | green-600 | green-50 | neutral-50 | green-700 | green-600 |
| warning | yellow-300 | yellow-100 | neutral-1000 | yellow-700 | yellow-400 |
| accent | blue-600 | blue-50 | neutral-50 | blue-700 | blue-600 |

Note: warning uses lighter steps than other groups because yellow is intrinsically high-lightness in OKLCH. warning.on-solid uses dark text (neutral-1000) because the solid fill is too bright for white text.

**Token mapping (dark mode):**

| Group | solid | light | on-solid | on-light | border |
|-------|-------|-------|----------|----------|--------|
| error | red-500 | red-950 | neutral-50 | red-300 | red-700 |
| success | green-500 | green-950 | neutral-50 | green-300 | green-700 |
| warning | yellow-400 | yellow-950 | neutral-1000 | yellow-300 | yellow-700 |
| accent | blue-600 | blue-950 | neutral-50 | blue-300 | blue-700 |

**Total: 20 status/accent tokens** (5 x 4), replacing 16 (4 x 4). Each token has an unambiguous job.

### 4. Overlay tokens for interactive states

Replace dedicated hover/pressed tokens with overlay tokens that work on any fill color. Components layer a semi-transparent overlay on top of their base fill using Figma's multiple-fill-per-frame capability.

**Overlay direction logic:**

| Fill type | Overlay direction | Rationale |
|-----------|-------------------|-----------|
| Solid/dark fills (`.solid`, `interactive.primary`) | Lighten (white-alpha) | Moves background toward light text — contrast preserved |
| Light/secondary fills (`.light`, `interactive.secondary`) | Darken (black-alpha) | Moves background toward dark text — contrast preserved |

**Overlay tokens:**

| Token | Value | Use |
|-------|-------|-----|
| `overlay.solid-hover` | white-alpha-8 | Hover on solid/dark fills |
| `overlay.solid-pressed` | white-alpha-16 | Pressed on solid/dark fills |
| `overlay.light-hover` | black-alpha-8 | Hover on light/secondary fills |
| `overlay.light-pressed` | black-alpha-16 | Pressed on light/secondary fills |

**Figma implementation:** Button background frame gets two fills. Fill 1 (bottom) is bound to the base color variable. Fill 2 (top) is bound to the overlay variable. Per state variant, the overlay variable swaps (default = transparent, hover = `overlay.solid-hover`, pressed = `overlay.solid-pressed`).

### 5. Disabled state treatment

Disabled strips color identity entirely. All button variants converge to the same neutral disabled appearance. Three dedicated tokens, values derived via OKLCH lightness targeting for ~2.5:1 contrast (readable but clearly inactive).

WCAG 2.1 SC 1.4.3 explicitly exempts disabled controls from contrast requirements. Best practice targets ~2.5:1.

| Token | Light value | Dark value | Role |
|-------|-------------|------------|------|
| `interactive.disabled` | neutral-200 | neutral-800 | Disabled fill for all button variants |
| `on-surface.disabled` | neutral-400 | neutral-600 | Disabled text/icon (universal) |
| `border.disabled` | neutral-200 | neutral-800 | Disabled stroke |
| `border.contrast` | neutral-1000 | neutral-50 | High-emphasis stroke (focused inputs, selected cards) |

This fixes the current neutral-300/neutral-300 collision between fill and text.

Research basis: all six surveyed systems (M3, Apple HIG, Spectrum, Carbon, Polaris, Lightning) strip color identity when disabled. Full comparison in `lab/research/2026-03-25-disabled-state-comparison.md`.

### 6. Slimmed interactive group

The `interactive` group drops hover/pressed/disabled sub-tokens (now handled by overlays and disabled tokens):

| Current | Proposed | Notes |
|---------|----------|-------|
| `interactive.primary` | `interactive.primary` | Unchanged |
| `interactive.primary-hover` | _(removed — use overlay)_ | Orphaned in Figma, flagged for cleanup |
| `interactive.primary-pressed` | _(removed — use overlay)_ | Orphaned in Figma, flagged for cleanup |
| `interactive.primary-disabled` | _(removed — use `interactive.disabled`)_ | Orphaned in Figma, flagged for cleanup |
| `interactive.secondary` | `interactive.secondary` | Unchanged |
| _(new)_ | `interactive.disabled` | New token for all disabled fills |

### 7. Intentional primitive overlaps are acceptable

Some tokens alias the same primitive step. This is by design — they serve different semantic roles and are scoped differently in Figma:

| Token A | Token B | Shared primitive | Why OK |
|---------|---------|------------------|--------|
| `surface.raised` | `interactive.secondary` | neutral-100 | Different Figma scopes; can diverge later |
| `surface.sunken` | `border.default` | neutral-200 | Different property types (fill vs stroke) |
| `interactive.disabled` | `border.disabled` | neutral-200 | Same value intentionally; different scopes |

## Figma migration strategy

The origin-sync plugin supports all the operations needed for a safe migration. The key constraint: **never delete Figma artifacts — update in place, flag orphans for manual review.**

### Phase 1: Additive changes (zero breakage)

Add new primitives and new semantic tokens. No existing references are touched.

| Action | Plugin behavior | Risk |
|--------|----------------|------|
| Add red, green, yellow primitive ramps | `upsertVariable()` creates new variables | None — purely additive |
| Add overlay tokens (solid-hover, solid-pressed, light-hover, light-pressed) | Creates new variables | None — purely additive |
| Add `interactive.disabled` token | Creates new variable | None — purely additive |
| Add `border.disabled` token | Creates new variable | None — purely additive |

### Phase 2: Renames via rename map (references preserved)

Use the plugin's reverse rename map to change token paths. Existing Figma references follow the rename automatically because the variable object stays the same — only its name and tokenPath pluginData change.

|----------------|---------------|----------------|
| `color/info/base` | `color/accent/solid` | Matched via reverseRenames → renamed in place |
| `color/info/subtle` | `color/accent/light` | Renamed in place |
| `color/info/text` | `color/accent/on-light` | Renamed in place |

## Open questions for stress-testing

1. **Exact anchor hexes for red, green, yellow ramps.** Need to select anchors that produce strong, clear signals at the steps semantics will use. Candidate process: pick a "hero" hex for each, convert to OKLCH, verify the ramp output at key steps (50, 300, 500, 600, 700, 950).

2. **Overlay opacity values.** 8%/16% are starting points from industry research. Need to validate these produce visible-enough feedback without breaking contrast. Test on: solid black button, solid red button, light gray secondary button, light blue accent surface.

3. **Dark mode overlay direction.** In dark mode, solid fills are lighter (e.g., red-500 vs red-600). Does white-alpha still produce good hover feedback on a mid-lightness fill? May need to test both directions.

4. **Warning on-solid text color.** Yellow is a light-value hue — solid yellow fills need dark text, not white. Need to verify neutral-1000 meets AA against the generated yellow-600.

5. **Figma scoping for overlay tokens.** Overlay tokens should scope to fill properties. Need to verify that scoping a semi-transparent variable to fills works correctly in Figma's variable system.

6. **Multiple fills + variable binding in Figma.** Validate that swapping a variable binding on fill 2 across component variants works reliably. Build a test button component to confirm.

7. **Platform implementation of overlays.** SwiftUI and Compose both support layered views with opacity. Confirm that Paul and Sergei can implement this pattern without friction.

## What this does NOT cover

- Component-level design changes (button component rebuild, input field states)
- Figma component library updates (separate workstream after tokens land)
- Output format updates (CSS, Tailwind, JSON exports)
- Dark mode overlay direction validation (open question above)
